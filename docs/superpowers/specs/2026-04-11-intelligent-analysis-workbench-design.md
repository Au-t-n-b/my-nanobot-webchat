# 智能分析工作台设计

**目标**

构建一个不绑定具体业务、可直接交付给团队参考的“全能力演示模块”。该模块进入会话时先通过项目级 `task_progress` 引导用户进入“智能分析工作台”，进入后完整展示当前 nanobot 已具备的关键能力：

- HITL 目标选择
- 多文件上传
- 上传后胶囊/卡片预览
- 右侧大盘流式动态更新进展与指标
- 多个 skill 的并行分析
- 串行的结论汇总与产物展示

**范围**

本轮先使用前端假数据和本地 `task_progress.json` 打通链路，不直接接真实登录/项目后端，但设计上预留“用户姓名 + 项目绑定 task_progress”的入口。后续真实业务只替换数据源，不重写交互骨架。

## 一、整体体验

用户进入网页后的体验分两层：

1. **项目引导层**
   登录后进入当前项目，系统读取该项目对应的 `task_progress`。新建或打开会话时，顶部和“项目总览”会直接显示“智能分析工作台”所处阶段，例如“模块待启动”或“资料已上传”。界面同时给出明确引导：打开该模块大盘继续执行。

2. **模块执行层**
   用户进入“智能分析工作台”后，左侧会话通过 GuidanceCard / ChoiceCard / FilePicker 完成交互，右侧大盘展示当前进展、黄金指标、产物总结，并在运行过程中以流式 patch 持续更新，而不是一次性跳变。

## 二、推荐方案

### 方案 A：在现有 `zhgk_module_case` 上继续泛化

优点：改动少，可复用现有 runtime。
缺点：模块语义仍然是工勘，不适合作为最终标准案例。

### 方案 B：新建独立的 `intelligent_analysis_workbench`

优点：完全围绕“全能力演示模块”设计，适合做团队标准案例；能把“项目引导层”和“模块执行层”明确分开。
缺点：需要新增模板和 flow。

### 方案 C：先做框架，再做示例模块

优点：最规范。
缺点：这一轮实现成本过高，会拖慢交付。

**推荐：方案 B。**

## 三、模块主线

采用“分析工作台型”的混合流程：

1. `guide`
   模块已被项目总览引导到当前阶段后，用户点击进入模块。模块打开右侧大盘，并在左侧给出启动说明。

2. `select_goal`
   通过 HITL ChoiceCard 让用户选择本次分析目标，例如：
   - 资料完整性分析
   - 风险识别分析
   - 交付结论生成
   - 综合分析

3. `upload_bundle`
   使用真实 FilePicker 上传多文件。上传完成后，会话中的卡片被替换成上传结果卡片，并显示文件胶囊预览。

4. `run_parallel_skills`
   模拟或实际调度 3 个分析 skill 并行执行，例如：
   - `file_inventory_skill`
   - `quality_scan_skill`
   - `risk_signal_skill`

5. `synthesize_result`
   在 3 个并行 skill 完成后，串行执行 `result_synthesis_skill`，汇总结论、生成建议，并更新产物区。

6. `finish`
   展示最终结论、指标最终态与产物总结。

## 四、项目级进度设计

`task_progress.json` 中为“智能分析工作台”预置一个项目阶段模块，默认进入会话时就可见，阶段定义如下：

1. 模块待启动
2. 分析目标已确认
3. 资料已上传
4. 并行分析进行中
5. 结论汇总中
6. 分析完成

该阶段进度负责“会话一进来就看见并被引导”的项目层体验；不负责展示模块运行细节。

后续接真实业务时，`task_progress` 的来源改为“登录用户 + 当前项目”的实际项目状态。

## 五、模块大盘设计

右侧大盘仍然保留统一的三块信息：

- 当前进展
- 黄金指标
- 产物总结

推荐节点：

- `stepper-main`
- `chart-donut`
- `chart-bar`
- `summary-text`
- `artifacts`

如果后续业务同事要换成更复杂图形，可以在保留稳定节点 id 的前提下，把柱状图/饼图替换成甘特图、分类统计图或自定义图形。

## 六、流式更新设计

当前“闪一下就更新”的根因是多数阶段只发一次最终态 patch。最终案例要改成分段流式：

1. 阶段开始：发送 `isPartial=true` 的 patch，将 stepper 节点切到 `running`
2. 中间推进：连续更新 summary、donut、bar 等节点
3. 阶段收敛：发送 `isPartial=false` 的稳定 patch，固化最终状态

并行分析阶段要求至少能看到 3 组分析状态先后或交错推进，而不是一次性全部完成。

## 七、上传与胶囊预览

会话里上传完成后，不再只显示“已收到文件”文本，而要替换为结构化卡片：

- 文件名
- 文件类型
- 可选的数量/大小
- 下一步动作提示

右侧产物区不承载“上传胶囊”，只承载最终分析产物；上传胶囊属于会话交互层。

## 八、真实业务的替换边界

后续接真实业务时，仅替换以下内容：

- 登录态和项目绑定的真实数据源
- `task_progress` 的真实项目阶段
- HITL 选项内容
- 上传文件类型与校验规则
- 并行分析 skill 的业务逻辑
- 结论汇总 skill 的业务逻辑

不建议替换以下骨架：

- 先项目引导、再进入模块
- 模块内 HITL -> 上传 -> 分析 -> 汇总 -> 产物 的主路径
- Dashboard 的稳定节点 id
- 项目层进度与模块层大盘分层的设计

## 九、测试策略

1. 后端测试
- `task_progress` 默认预置“智能分析工作台”
- 模块 flow 在各 action 上输出正确 patch
- 并行分析阶段能发出多次 partial patch

2. 前端测试
- 项目总览能显示工作台阶段
- 模块进入后能展示流式进展
- 上传完成后会话卡片能展示结构化预览

3. 集成验证
- 新建会话即可看到智能分析工作台阶段
- 点击入口打开大盘
- 完整跑通 HITL、上传、并行分析、结论汇总、产物展示
