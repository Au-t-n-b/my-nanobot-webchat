# 模块平台协议设计

**目标**

构建一套可复用的“模块平台 + 协议 + 模板”，供各业务模块开发同事直接参考接入。平台层提供统一能力、统一协议和标准模板；模块开发者只负责定义本模块的业务逻辑、HITL 选项、上传规则、指标呈现和项目阶段任务。

该设计的目标不是交付一个写死的案例页面，而是交付一个长期可扩展的模块开发标准，使多个模块最终都能稳定呈现在项目总览中，并在模块执行过程中保持会话、大盘、任务进度三者同步。

**范围**

本轮定义以下内容：

- 平台层与模块层的职责边界
- 模块目录和接入协议
- HITL、文件上传、文件预览、黄金指标呈现的配置边界
- `task_progress` 的定义方式与自动更新机制
- 项目总览与模块大盘的数据分层

本轮不要求接真实登录和项目后端，但设计必须兼容后续“用户 + 项目绑定”的正式接入。

## 一、推荐方案

### 方案 A：强模板

所有模块都沿用完全相同的页面结构和图表形式。

优点：开发快，统一性强。
缺点：无法适应真实业务差异，开发者会被迫绕模板写大量特例。

### 方案 B：完全自由

平台只提供基础通信能力，每个模块自行决定交互、上传和大盘形态。

优点：灵活。
缺点：模块间风格和协议会迅速失控，项目总览与运行时维护成本很高。

### 方案 C：平台能力统一，模块内容可配置

平台统一提供公共协议、通用组件、运行时能力和标准模板；模块开发者通过配置和业务 flow 定义自己的内容。

优点：既能保证接入一致性，又能保留业务自由度，最适合作为多模块长期标准。

**推荐：方案 C。**

## 二、职责边界

### 平台层负责

1. 统一协议
   - `chat_card_intent`
   - `module_action`
   - `task_progress`
   - `SkillUiDataPatch`
   - `open_preview`

2. 通用组件
   - GuidanceCard
   - ChoiceCard
   - FilePicker
   - Stepper
   - GoldenMetrics
   - BarChart
   - DonutChart
   - ArtifactGrid
   - 嵌入式 Web 节点

3. 运行时能力
   - 模块会话状态保存
   - 文件上传与路径落盘
   - 右侧分栏预览
   - 大盘 patch 推送
   - 产物挂载
   - `task_progress` 自动更新与广播
   - 模块配置 Schema 校验

4. 标准模板
   - 目录骨架
   - 默认 flow
   - 标准节点 id
   - 接入示例

### 模块开发者负责

1. 定义 HITL 选项
2. 定义上传文件规则与目标路径
3. 定义黄金指标及其表现形式
4. 定义业务 flow
5. 定义 `task_progress` 任务结构和 action 映射
6. 定义产物总结内容

## 三、目录与模板结构

每个模块统一采用以下目录结构：

- `module.json`
- `SKILL.md`
- `data/dashboard.json`
- `references/flow.md`
- `input/`
- `output/`

其中：

- `module.json`：模块元信息、flow、能力配置、任务进度配置
- `SKILL.md`：模块业务逻辑说明
- `dashboard.json`：模块大盘基础骨架
- `references/flow.md`：给模块开发者的实现参考
- `input/`：上传文件落盘目录
- `output/`：最终产物目录

平台层必须为 `module.json` 和 `dashboard.json` 提供 JSON Schema 强校验。模块加载时一旦发现缺失必填字段、字段名拼写错误、节点结构不合法或关键 id 配置错误，应在控制台与运行时入口直接给出明确报错，而不是带着畸形配置进入运行时。

建议至少校验以下内容：

- `module.json` 是否包含 `moduleId`、`flow`、`docId`、`dataFile`
- 上传节点是否包含 `save_relative_dir`
- `task_progress` 映射是否完整
- `dashboard.json` 是否符合节点 schema
- 是否存在必须保留的关键节点 id：`stepper-main`、`summary-text`、`uploaded-files`、`artifacts`

## 四、数据分层原则

平台中存在两类不同层级的数据，必须分开：

1. **项目层**
   由 `task_progress` 驱动。
   用于项目总览中展示模块列表、阶段状态、甘特/阶段进度。

2. **模块层**
   由 `dashboard.json + SkillUiDataPatch` 驱动。
   用于模块大盘中展示实时进展、黄金指标、文件栏和产物总结。

**原则：**

- 项目总览不读取模块内部 patch
- 模块大盘不承担项目级进度计算
- 两者通过模块 id 关联，但数据职责分离

## 五、HITL 协议

HITL 由模块开发者自定义，平台只提供统一协议。

模块开发者可自由定义：

- 选项内容
- 选项值
- 默认选项
- 下一步 action

平台统一要求：

- 所有 HITL 都通过 `chat_card_intent -> module_action`
- 所有选择结果进入模块会话状态
- 选择完成后可驱动 Stepper、黄金指标和 `task_progress` 更新

## 六、文件上传协议

文件上传必须作为平台层公共能力存在，不能由模块开发者各自实现一套。

### 平台提供

- `ask_for_file(...)`
- `FilePicker`
- 上传接口
- 文件落盘
- 返回可预览路径
- 上传结果状态写入模块会话

### 模块开发者配置

每个上传节点至少定义：

- `purpose`
- `accept`
- `multiple`
- `save_relative_dir`
- `next_action`

其中 `save_relative_dir` 由模块开发者定义，例如：

- `skills/contract_review/input`
- `skills/risk_scan/input/evidence`
- `skills/finance_module/input/raw`

平台必须保证文件上传后直接落到该路径下。

平台运行时还应统一把上传结果规范为：

- `upload`
- `uploads[]`

以保证单文件和多文件场景都能共用同一套预览与大盘同步协议。

## 七、上传结果与预览协议

上传结果不能只以一句文本存在，必须形成结构化状态。

建议平台统一维护：

- `uploads[]`

每个文件至少包含：

- `fileId`
- `name`
- `logicalPath`
- `savedDir`
- `uploadedAt`

### 上传完成后的统一表现

1. 会话中出现“已上传文件栏”
2. 文件以胶囊按钮形式展示
3. 模块大盘底部也展示同一份“已上传文件栏”
4. 点击胶囊直接通过 `open_preview(path)` 在右侧分栏预览

建议模块大盘中固定一个节点 id：

- `uploaded-files`

该区域应优先复用现有的 `ArtifactGrid` 组件，不再单独实现一套输入文件胶囊组件。建议在组件层增加轻量模式参数，例如：

- `mode="input"`
- `mode="output"`

两种模式共享同一套胶囊渲染、点击预览和列表布局逻辑，仅在颜色主题、标题文案或少量交互细节上区分，保证平台层实现保持 DRY。

## 八、黄金指标协议

黄金指标的内容和呈现方式必须由模块开发者定义，平台不能写死成某一种图表。

平台应支持以下表现形式：

- 指标卡 + 横向条形
- 柱状图
- 环图/饼图
- 嵌入式网页
- 混合形式

平台层只提供可选组件与 patch 能力，模块开发者在 `dashboard.json` 中选择使用何种节点。

**建议：**

- 平台模板默认提供“指标卡 + 柱状图”的标准版
- 允许模块替换为环图或网页嵌入
- 保持关键节点 id 稳定，便于 patch 命中

## 九、task_progress 定义与自动更新

`task_progress` 的结构由模块开发者定义，但更新执行由平台统一承接。

### 模块开发者负责定义

在 `module.json` 中定义：

- `moduleId`
- `moduleName`
- 任务列表
- action 到任务完成状态的映射

例如：

- `guide` 完成后，标记“模块待启动”
- `select_goal` 完成后，标记“分析目标已确认”
- `upload_bundle` 完成后，标记“资料已上传”
- `run_parallel_skills` 完成后，标记“并行分析进行中”
- `finish` 完成后，标记“分析完成”

### 平台负责自动更新

每次 `module_action` 成功执行后，平台运行时自动：

1. 读取该模块的 `task_progress` 配置
2. 根据当前 action 找到需完成的任务
3. 更新 `task_progress`
4. 推送 `TaskStatusUpdate`
5. 刷新项目总览

**原则：**

- 模块开发者定义任务结构和映射
- 平台负责执行更新和广播
- 不允许各模块手写分散的任务进度更新逻辑

## 十、模块标准主线

作为团队参考模板，建议所有模块至少具备如下主线能力：

1. `guide`
   模块进入与说明

2. `select_xxx`
   至少一个 HITL 决策点

3. `upload_xxx`
   至少一个上传节点，且支持开发者自定义路径

4. `after_upload`
   上传结果进入状态，并展示文件胶囊与预览

5. `run_xxx`
   执行模块主逻辑，并流式 patch 大盘

6. `finish`
   生成最终结论和产物

## 十一、项目总览最终形态

项目总览统一展示：

- 模块列表
- 模块状态
- 模块阶段完成度
- 甘特图或阶段条
- 点击进入对应模块大盘

项目总览不展示模块内部上传文件，不承担模块细节展示。

模块细节统一进入模块大盘，包括：

- Stepper
- 黄金指标
- 已上传文件栏
- 产物总结

## 十二、接入清单

每个模块接入前应完成以下检查：

1. 是否注册到 `task_progress`
2. 是否定义至少一个 HITL
3. 是否定义上传规则和上传路径
4. 是否上传后展示会话文件胶囊
5. 是否上传后展示大盘底部文件栏
6. 是否支持右侧分栏预览
7. 是否支持大盘 patch
8. 是否支持黄金指标自定义表现形式
9. 是否支持任务完成后自动更新 `task_progress`
10. 是否在 `finish` 后生成产物

## 十三、推荐母版

当前推荐将 `intelligent_analysis_workbench` 作为标准母版模块。

该母版应重点演示以下平台能力：

- 项目总览接入
- 自动打开模块大盘
- HITL 选择
- 文件上传与落盘
- 会话文件胶囊
- 大盘底部文件栏
- 右侧分栏文件预览
- 黄金指标多形态支持
- 流式 patch
- `task_progress` 自动更新
- 最终产物挂载

## 十四、测试策略

1. 平台层测试
   - `module.json` 与 `dashboard.json` 是否通过 Schema 校验
   - 配置错误时是否给出明确报错
   - 上传路径是否按模块配置落盘
   - `open_preview` 是否可预览上传文件
   - `task_progress` 是否按 action 自动更新
   - patch 是否能稳定更新模块大盘

2. 模块层测试
   - HITL 选项是否按模块定义呈现
   - 黄金指标是否按模块定义呈现
   - 上传后文件栏是否在会话和大盘底部同时出现

3. 集成验证
   - 多个模块是否可同时出现在项目总览
   - 模块执行后是否能驱动总览与模块大盘同步变化
   - 最终产物与上传文件预览是否都可正常打开
